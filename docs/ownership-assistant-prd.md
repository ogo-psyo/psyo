# PRD: Псё как ассистент по владению и воспитанию собаки

## 1. Summary

Псё — приложение для владельца собаки, где аватар остаётся catchy-механикой входа, но основной продукт — персональный ассистент по владению, воспитанию, здоровью, социализации и повседневным решениям вокруг собаки.

Короткая формула:

> **Аватар привлекает. Паспортичка даёт контекст. Ассистент и напоминания удерживают. Карта, клиники, ветшопы и вишлисты монетизируют и расширяют пользу.**

## 2. Problem Statement

Владельцу собаки приходится держать в голове слишком много разрозненных вещей:

- документы, прививки, обработки, лекарства;
- поведение, триггеры, воспитание;
- где гулять и с кем можно знакомиться;
- когда идти к врачу, грумеру, в магазин;
- что покупать, что подходит конкретной собаке;
- как объяснить особенности собаки догситтеру, врачу или другому владельцу.

Обычные pet apps часто превращаются либо в медицинский трекер, либо в социальную карту, либо в магазин. Псё должно стать **единым контекстным ассистентом владельца**, а не одной узкой утилитой.

## 3. Goals / Success Criteria

### Product goals

- Сделать профиль собаки полезным после первого вау-эффекта.
- Превратить “карточку питомца” в рабочий контекст для ассистента.
- Дать владельцу причины возвращаться: напоминания, советы, карта, социализация, покупки.
- Подготовить естественные партнёрские слои: клиники, ветшопы, подборки, вишлисты.

### Success signals

- Пользователь заполняет не только имя/породу, но и 3+ практических блока: здоровье, поведение, социализация, напоминания.
- Пользователь создаёт хотя бы одно напоминание или wishlist item.
- Пользователь задаёт ассистенту вопрос по поведению/уходу.
- Пользователь возвращается в течение 7 дней из-за care/reminder сценария.

## 4. Non-Goals / Out of Scope for near MVP

- Полноценная ветеринарная диагностика.
- Замена врача, кинолога или тренера.
- Точная GPS-слежка за собакой или владельцем.
- Агрессивный marketplace до доказанной пользы.
- Сложная соцсеть с чатами на первом этапе.
- Реальные интеграции с клиниками/ветшопами до ручной проверки спроса.

## 5. Users / Actors

- **Основной владелец** — создаёт профиль, ведёт здоровье, задаёт вопросы ассистенту.
- **Семья / второй владелец** — смотрит карточку, получает напоминания, знает правила ухода.
- **Догситтер / выгульщик** — получает ограниченную паспортичку: триггеры, еда, здоровье, контакты.
- **Ветеринар / клиника** — потенциально получает документы и историю.
- **Ветшоп / партнёр** — потенциально даёт подборки под профиль собаки.

## 6. Product Pillars

### 6.1 Паспортичка

База данных о собаке:

- имя, фото, аватар;
- порода / группа / метис;
- возраст, пол, стерилизация;
- вес, размер, шерсть, особенности внешности;
- микрочип;
- ветклиника / врач;
- питание, аллергии, лекарства;
- заметки для врача/догситтера.

**Роль:** контекст для всех следующих функций.

### 6.2 Карта / зоны на карте

Не “где моя собака в реальном времени”, а безопасный слой мест:

- примерная зона проживания/прогулок;
- любимые маршруты;
- dog-friendly места;
- зоны риска: агрессивные собаки, стройка, клещи, салюты;
- клиники, грумеры, ветшопы;
- nearby dogs только с privacy controls.

**Правило:** никакого точного GPS по умолчанию.

### 6.3 Социализация

Профиль совместимости:

- как подходить: можно / сначала спросить / лучше не подходить;
- дружит ли с собаками, детьми, кошками;
- триггеры: самокаты, гром, кобели, поводок, еда;
- стиль игры: догонялки, апорт, борьба, нюховые игры;
- одиночество и тревожность;
- публичная короткая подсказка на карточке.

**Роль:** снижать риск плохих знакомств и делать карту полезной.

### 6.4 Ассистент

Главный долгосрочный слой продукта.

Сценарии:

- воспитание: поводок, подзыв, лай, одиночество, туалет;
- уход: что делать сегодня, сезонные риски, груминг;
- здоровье: triage симптомов, красные флаги, когда к врачу;
- подготовка: поездка, перелёт, передержка, визит к врачу;
- подбор: корм, игрушки, амуниция, курсы, клиники.

Безопасная рамка:

- ассистент не ставит диагноз;
- явно отделяет “можно наблюдать” от “нужно к врачу”; 
- использует профиль собаки, но не притворяется ветеринаром.

### 6.5 Напоминания

Поводы вернуться:

- вакцинация;
- обработка от клещей/глистов;
- лекарства;
- корм купить;
- груминг;
- тренировки;
- вес/замеры;
- визит к врачу;
- сезонные риски.

MVP: ручные напоминания + шаблоны. Позже: ассистент предлагает напоминание из диалога.

### 6.6 Интеграции с клиниками / ветшопами / подборками

Партнёрский слой должен появляться после пользы, а не вместо неё.

Возможности:

- сохранить основную клинику;
- подбор ближайших клиник/грумеров/ветшопов;
- подборки товаров под размер, возраст, аллергию, поведение;
- “купить снова” для расходников;
- рекомендации ассистента с объяснением почему подходит.

Риск: если начать с магазина, продукт станет рекламным. Поэтому marketplace должен быть контекстным и полезным.

### 6.7 Вишлисты для собаки

Сценарии:

- список покупок владельца;
- подарочный wishlist для друзей;
- регулярные расходники;
- “хочу попробовать”;
- ассистент добавляет item после рекомендации.

Поля item:

- название;
- категория;
- причина;
- приоритет;
- ссылка;
- статус: хочу / куплено / не подошло.

## 7. Proposed UX Shape

### First session

1. Вау-хук: загрузить фото → создать аватар.
2. Мини-паспортичка: имя, порода/группа, возраст, размер.
3. Быстрый вопрос: “Что важно помнить о собаке?”
4. Предложение выбрать первый якорь:
   - напоминание;
   - социализация;
   - вопрос ассистенту;
   - wishlist.

### Home screen after setup

- карточка собаки;
- “что сегодня важно”;
- ближайшие напоминания;
- быстрые вопросы ассистенту;
- карта/зоны;
- wishlist / подборки.

## 8. Functional Requirements

### MVP+ functional requirements

- User can create dog profile from avatar hook.
- User can fill passport fields.
- User can set socialization rules.
- User can set care statuses and next care date.
- User can see assistant scenario cards.
- User can see map privacy framing.
- User can understand future wishlist/partner layer from UI.

### Later requirements

- Real reminders.
- Auth and persistent DB.
- AI assistant with dog-profile context.
- Shareable limited passport.
- Wishlist CRUD.
- Partner catalog and recommendations.
- Real map layers and moderated zones.

## 9. UX / Design Requirements

- Аватар не должен доминировать как “весь продукт”. Он входной крючок.
- Основная фраза продукта: **ассистент по владению и воспитанию пса**.
- Профиль должен выглядеть не как бюрократическая анкета, а как “память владельца”.
- Глубокие поля показывать слоями: сначала essential, потом behavior/care/docs.
- Все медицинские подсказки — с безопасной формулировкой.
- Карта — privacy-first.

## 10. Acceptance Criteria for Current Iteration

- Hero clearly says product is ownership/training assistant.
- Avatar is framed as catchy hook, not core product.
- UI contains pillars: паспортичка, карта/зоны, социализация, ассистент, напоминания, клиники/ветшопы/подборки, вишлисты.
- Assistant section explains context-aware assistant.
- Build passes.

## 11. Risks / Edge Cases

- **Too much form upfront:** solve with progressive disclosure.
- **Medical liability:** triage only, red flags, “go to vet” boundaries.
- **Privacy:** approximate zones, no exact location by default.
- **Marketplace trust risk:** recommendations must explain relevance; avoid ad feel.
- **Breed stereotyping:** breed informs, but actual behavior/profile overrides.
